1. Objetivo del Laboratorio 
Garantizar que cualquier cambio futuro en el código del motor de evaluación de riesgo crediticio de 
FinTech Nova sea validado automáticamente antes de llegar a producción. Esto se logra mediante 
un pipeline de Integración y Despliegue Continuo (CI/CD) en GitHub Actions que, en cada push a 
la rama main, descarga el código, instala las dependencias, ejecuta una auditoría de seguridad 
sobre ellas, y construye la imagen Docker final únicamente si todo lo anterior es exitoso. 
Este documento cubre los tres criterios de evaluación del Laboratorio 4: (1) el pipeline CI/CD 
funcionando en verde, (2) el diagrama arquitectónico final actualizado, y (3) los puntos clave para 
la sustentación técnica. Adicionalmente, incluye una demostración opcional (sección 3) donde se 
provoca intencionalmente un fallo de seguridad para comprobar que el pipeline lo detecta y 
bloquea automáticamente. 
2. Pipeline de Integración y Despliegue Continuo 
2.1 Disparador y Estructura 
El pipeline se ubica en .github/workflows/ci-cd-pipeline.yml y se activa automáticamente con cada 
push a la rama main (y también puede lanzarse manualmente con workflow_dispatch). Está 
compuesto por un solo job (build-and-security) que ejecuta seis pasos en una máquina virtual 
Ubuntu efímera: descarga del código, configuración de Python 3.11, instalación de dependencias, 
auditoría de seguridad con pip-audit, ejecución de pruebas automáticas con pytest, construcción de 
la imagen Docker, y una verificación final (smoke test) de que la imagen levanta correctamente.

        uses: actions/checkout@v4 
  
      - name: Configurar Python 3.11 
        uses: actions/setup-python@v5 
        with: 
          python-version: '3.11' 
          cache: 'pip' 
  
      - name: Instalar dependencias 
        working-directory: fintech-nova-api 
        run: | 
          python -m pip install --upgrade pip 
          pip install -r requirements.txt -r requirements-dev.txt 
  
      - name: Auditoría de seguridad con pip-audit 
        working-directory: fintech-nova-api 
        run: | 
          pip install pip-audit 
          pip-audit --requirement requirements.txt --format=columns 
  
      - name: Ejecutar pruebas automáticas 
        working-directory: fintech-nova-api 
        run: pytest test_main.py -v --tb=short 
  
      - name: Construir imagen Docker 
        run: docker build -t fintech-nova:${{ github.sha }} fintech-nova-api/ 
  
      - name: Verificar imagen Docker (Smoke Test) 
        run: | 
          docker run -d --name fintech-test -p 8000:8000 fintech-nova:${{ github.sha }} 
          sleep 5 
          curl --fail http://localhost:8000/status || (docker logs fintech-test && exit 1) 
          docker stop fintech-test && docker rm fintech-test 
 
2.3 Auditoría de Seguridad de Dependencias (pip-audit) 
Como parte del enfoque DevSecOps (Shift-Left), el pipeline ejecuta pip-audit antes de construir la 
imagen Docker. Esta herramienta compara cada dependencia del proyecto (fastapi, uvicorn, psutil, 
pydantic) contra las bases de datos de vulnerabilidades OSV y NVD. Si se detecta una 
vulnerabilidad conocida, el pipeline se detiene automáticamente y no se construye ninguna imagen, 
evitando que código inseguro llegue a producción.

2.4 Pruebas Automáticas (pytest) 
Se desarrolló una suite de pruebas (test_main.py) que valida la lógica de negocio del motor de 
riesgo: el rechazo automático a menores de edad, la aprobación cuando el score (ingresos menos 
deudas) supera el umbral, el estado 'En Revision' en casos intermedios, el rechazo de payloads 
inválidos con código 422, y la consulta del historial financiero. El pipeline ejecuta esta suite 
automáticamente antes de construir la imagen Docker; si alguna prueba falla, el build se detiene.

2.5 Verificación de la Imagen Docker (Smoke Test) 
Una vez construida la imagen, el pipeline la ejecuta temporalmente y consulta el endpoint /status 
mediante curl. Si la API no responde correctamente, el paso falla y se muestran los logs del 
contenedor para diagnóstico. Esto confirma que la imagen no solo se construye, sino que 
efectivamente arranca y sirve peticiones — un nivel de verificación adicional sobre la sola 
construcción de la imagen.

3. Demo Opcional: DevSecOps en Acción (Shift-Left) 
Para demostrar de forma tangible que el control de seguridad funciona, se realizó una prueba 
controlada: se introdujo intencionalmente una dependencia con una vulnerabilidad conocida 
(requests==2.6.0, asociada al CVE GHSA-j8r2-6x86-q33q) en requirements.txt y se subió al 
repositorio. El objetivo era observar si el pipeline detectaba y bloqueaba el problema 
automáticamente, sin intervención humana. 
Resultado: el paso de pip-audit falló inmediatamente con código de salida distinto de cero, y 
GitHub Actions detuvo el pipeline antes de llegar a las pruebas, la construcción de la imagen 
Docker y el smoke test. Ningún código vulnerable llegó a empaquetarse. Posteriormente se 
restauró la versión segura de la dependencia y se confirmó que el pipeline volvió a completarse 
exitosamente.

5. Puntos Clave para la Sustentación 
Guía de respuestas para explicar con dominio técnico las decisiones tomadas en este laboratorio 
frente al panel evaluador. 
• ¿Por qué el pipeline se activa solo con push a main? Porque main es la rama de producción; 
las ramas de desarrollo pueden tener código en progreso que no debe pasar 
automáticamente los controles de calidad. 
• ¿Por qué se ejecuta pip-audit antes de construir la imagen Docker? Para aplicar el principio 
Shift-Left: detectar vulnerabilidades en las dependencias lo antes posible en el ciclo, evitando 
construir y desplegar una imagen insegura. 
• ¿Qué pasaría si una dependencia tuviera una vulnerabilidad conocida? pip-audit devolvería 
un código de salida distinto de cero, GitHub Actions marcaría ese paso como fallido, y el 
pipeline se detendría antes de llegar a las pruebas y a la construcción de la imagen Docker. 
Esto fue demostrado en la prueba controlada de la sección 3. 
• ¿Por qué se agregaron pruebas automáticas (pytest) además de la auditoría de seguridad? 
La auditoría revisa vulnerabilidades en las dependencias externas, pero no garantiza que la

lógica de negocio funcione correctamente. pytest valida que las reglas de evaluación de 
riesgo (rechazo a menores de edad, aprobación por score, validación de payloads) sigan 
comportándose como se espera ante cualquier cambio futuro en el código. 
• ¿Por qué se agregó un smoke test después de construir la imagen Docker? Porque que la 
imagen se construya sin errores no garantiza que la aplicación arranque correctamente 
dentro del contenedor. El smoke test ejecuta la imagen real y consulta un endpoint para 
confirmar que el servicio efectivamente responde antes de considerarlo listo para desplegar. 
• ¿Por qué la imagen Docker se etiqueta con ${{ github.sha }}? Para tener trazabilidad exacta 
entre cada imagen construida y el commit de código que la originó, lo que permite hacer 
rollback a una versión anterior específica si algo falla en producción. 
• ¿Qué corregimos respecto a la primera versión del workflow? Se eliminó una duplicación 
accidental de las claves on y jobs (que invalidaba el archivo YAML) y se corrigieron las rutas 
de requirements.txt y del contexto de build, ya que el código de la API vive en la subcarpeta 
fintech-nova-api/. 
6. Conclusiones 
• El pipeline se activa automáticamente en cada push a main, sin intervención manual, 
cumpliendo el principio de integración continua. 
• La auditoría de seguridad con pip-audit actúa como un control automático que bloquea 
dependencias vulnerables antes de llegar a producción (Shift-Left), comprobado mediante 
una prueba controlada. 
• La suite de pruebas con pytest valida automáticamente la lógica de negocio del motor de 
riesgo en cada cambio, evitando regresiones. 
• El smoke test confirma que la imagen Docker no solo se construye, sino que efectivamente 
arranca y responde correctamente. 
• La imagen Docker generada queda etiquetada con el SHA del commit, permitiendo 
trazabilidad exacta entre código e imagen, y facilitando un rollback inmediato si fuera 
necesario. 
• Con este laboratorio se completa la última capa de madurez del proyecto: el código de 
FinTech Nova ahora se valida, audita, prueba y empaqueta automáticamente antes de 
cualquier despliegue.
